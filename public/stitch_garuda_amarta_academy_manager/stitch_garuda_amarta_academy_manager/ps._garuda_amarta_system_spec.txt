# PS. Garuda Amarta Management System Specification

## 1. Database Schema (Migrations)

### Roles
- id
- nama_role (Super Admin, Bendahara, Pelatih, Murid)

### Rantings
- id
- nama_ranting
- keterangan

### Users
- id
- role_id (FK)
- ranting_id (FK, nullable)
- name
- email
- password
- phone

### Kurikulums
- id
- judul
- konten (Text/HTML)
- file_path (nullable)

### Nilais
- id
- murid_id (FK users)
- pelatih_id (FK users)
- ranting_tujuan_id (FK rantings)
- nilai_fisik (integer)
- nilai_teknik (integer)
- status_lulus (boolean)
- tanggal

### Ranting Latihans
- id
- pelatih_id (FK users)
- nama_ranting_tempat_latihan

### Laporan Royaltis
- id
- ranting_latihan_id (FK)
- bulan_tahun (string, e.g., '2023-10')
- pendapatan_kotor (decimal)
- potongan_kas (decimal)
- potongan_royalti (decimal)
- total_disetor (decimal)

### Events & Registrations
- events (id, nama_acara, tanggal_pelaksanaan, deskripsi)
- event_registrations (id, event_id, murid_id, file_berkas_path, status['Menunggu', 'ACC', 'Ditolak'])

### Master Data Tables
- master_pengurus
- master_sabuk_honor
- master_royalti
- master_perlengkapan
- master_nilai_prestasi

## 2. Business Logic

### Royalty Calculation Logic
When submitting `pendapatan_kotor`:
- If < 500,000: `potongan_kas` = 0, `potongan_royalti` = 50,000.
- If >= 500,000:
    - `potongan_kas` = `pendapatan_kotor` * 0.03 (3%).
    - `potongan_royalti` scaling:
        - <= 1,000,000: 100,000
        - <= 1,500,000: 150,000
        - (Pattern continues: +50k for every 500k bracket)
- `total_disetor` = `potongan_kas` + `potongan_royalti`.

### Dynamic Watermark (Pelatih View)
Overlay for Kurikulum content: "{User Name} - {User ID} - {Current Timestamp}" repeated diagonally.

### Role-Based Access
- Super Admin: Full system access.
- Bendahara: Financial modules & royalty views.
- Pelatih: Grades, Certificates, Royalty Reports, Curriculum (Watermarked).
- Murid: Profile, Grades, Event Registration.
